# secure-sdlc-requirements-threat-modeling

1. Confirm product types, criticality, delivery model, lifecycle, boundaries, owner, reviewer, and approver.
2. Map security activities, evidence, owners, entry criteria, exit criteria, exceptions, and escalation to lifecycle stages.
3. Derive testable requirements from product context, threats, approved policy, and architecture.
4. Model assets, actors, data flows, trust boundaries, dependencies, threats, abuse cases, and failure modes.
5. Link mitigations to requirements, owners, and validation methods.
6. Route high-impact outputs to `independent-appsec-reviewer`.

Stop for release approval, legal requirement invention, active exploitation, or risk acceptance.
