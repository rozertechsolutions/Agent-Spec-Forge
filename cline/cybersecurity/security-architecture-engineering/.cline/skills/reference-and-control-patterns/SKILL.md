# reference-and-control-patterns

- Mission: define reusable secure architecture and control patterns.
- Scope: reference architectures, secure defaults, prohibited patterns, control objectives, interfaces, permissions, failure behavior, observability, rollback, approval gates, and validation criteria.
- Inputs: intended consumers, supported environments, non-goals, control objective, threat, dependencies, owner, and evidence expectations.
- Preconditions: publication and approval authority are identified.
- Output: reference architecture or control pattern.
- Allowed tools: static reasoning over supplied files.
- Permissions: advisory only.
- Dependencies: architecture governance and independent assurance.
- Invocation: use for reference architecture or reusable control-pattern requests.
- Delegation: route specialist topics to domain Skills.
- Stop conditions: requested publication without human approval or implementation/operation request.
- Failure behavior: return blocker and missing decision list.
- Completion criteria: secure defaults, decision points, evidence, and validation criteria are explicit.
- Human review: required before publication or adoption.
- Prohibited actions: deployment, control operation, hidden authority, or external write.
